查詢指定地址是否屬於該 VASP。

## Request

```json
{
  "address": "0x742d35Cc6634C0532925a3b844Bc9e7595f8fE21",
  "network": "ethereum",
  "chain_id": "eip155:1",
  "asset_id": "eip155:1/slip44:60",
  "memo": null,
  "request_id": "req_abc123"
}
```

### Request Fields

| 欄位 | 類型 | 必填 | 說明 |
|------|------|------|------|
| address | string | Y | 要查詢的區塊鏈地址 |
| network | string | N | 區塊鏈網路短名別名（如 `ethereum`）；過渡期保留，完整切換 CAIP-only 規劃於 v2.3 |
| chain_id | string | Y | CAIP-2 chain_id（如 `eip155:1`），鏈的正規識別碼 |
| asset_id | string | N | CAIP-19 asset_id（用於多資產地址或需精確指定資產時） |
| memo | string | N | **[v2.0 新增]** Memo / Destination Tag（適用於 XRP, XLM, ATOM, EOS 等需要 Memo/Tag 的鏈） |
| request_id | string | Y | 請求唯一識別碼（用於冪等性） |

<Accordion title="[v2.0 新增] 關於 Memo / Destination Tag">
部分區塊鏈（如 XRP、XLM、ATOM、EOS）使用共用地址搭配 Memo 或 Destination Tag 來區分不同用戶。對於這類鏈，僅查詢地址（address）可能不足以確認受益人身份，必須同時提供 `memo` 欄位。

- 若該鏈不使用 Memo/Tag 機制，此欄位可為 `null` 或省略
- 受益方 VASP 在驗證地址時，應同時比對 address 與 memo 的組合
</Accordion>

## Response

<Tabs>
  <Tab title="地址屬於該 VASP">
```json
{
  "request_id": "req_abc123",
  "owned": true,
  "vasp_id": "VASP_B",
  "vasp_name": "交易所 B",
  "address": "0x742d35Cc6634C0532925a3b844Bc9e7595f8fE21",
  "memo": null,
  "verified_at": "2024-01-21T10:00:00Z"
}
```
  </Tab>
  <Tab title="地址不屬於該 VASP">
```json
{
  "request_id": "req_abc123",
  "owned": false,
  "address": "0x742d35Cc6634C0532925a3b844Bc9e7595f8fE21",
  "memo": null,
  "verified_at": "2024-01-21T10:00:00Z"
}
```
  </Tab>
</Tabs>

### Response Fields

| 欄位 | 類型 | 必填 | 說明 |
|------|------|------|------|
| request_id | string | Y | 對應請求的識別碼 |
| owned | boolean | Y | 該地址是否屬於此 VASP |
| vasp_id | string | N | VASP 識別碼（owned=true 時） |
| vasp_name | string | N | VASP 名稱（owned=true 時） |
| address | string | Y | 查詢的地址 |
| memo | string | N | **[v2.0 新增]** 對應的 Memo / Destination Tag |
| verified_at | string | Y | 查詢時間（ISO 8601） |

## 驗證結果暫存機制 [v2.2 新增]

<Note>
**第四次技術會議決議（提案 4）**

發起方 VASP 成功驗證接收方地址（`owned=true`）後，**可暫存該驗證結果**以減少重複查詢，規範如下：

- **暫存有效期（TTL）**：規格統一暫定 **30 天**
- **強制失效條件**：發起方定期呼叫 `GET /vasp/info` 時，若偵測到受益方 `config_version` 遞增（代表該 VASP 設定變更，例如**錢包地址重新配發**），應**立即作廢**該 VASP 相關的所有地址暫存
- 各 VASP 若需延長或縮短 TTL，可依自身實作與風險判斷調整；但規格層級先統一規定 30 天上限
</Note>

</Note>

<Warning>
**不得將「timeout 即放行」訂為共同標準。** 若允許「等待逾時無回覆即視為非境內同業地址並放行」成為標準做法，將產生誘因扭曲——業者只要消極維運、放任 endpoint timeout，即可實質規避境內 Travel Rule 確認義務，使整套機制形同架空，難以通過主管機關與內控查核。
</Warning>

**A. 對方 VASP timeout / 無回應**

- 各 VASP **有義務確保自身 endpoint 的可用性、健康度與即時監控**，並對異常狀況訂有維運與補救措施
- 當 `GET /health` 回覆 `healthy` 時，其餘 API 結果**必須於常規時間內返回**（常規回應時間之業界建議值見 [GET /health](/spec/endpoints/health)）
- 業者實作發生問題時，應**第一時間向公會反應**
- 對方 endpoint 離線時，發起方的後續處理由各家依內控制度、風險承受度與合規判斷自行設計（**不強制一致**）。法遵建議做法：對持續無回覆者每日定時重試；若最終結果不一致或始終無結果，應據以**調整該客戶風險等級或要求補正**

**B. 多家 VASP 對同一 address / memo 均回 `owned=true`**

正常情況下一個 `address`（+`memo`）至多應屬於一家 VASP。若出現多家回報 `owned=true`，屬異常情境。此情境已於 LINE 群組討論，**處理方式留待各家 VASP 依內控政策自行處理**，規格不強制統一做法。
