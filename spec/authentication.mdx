## 傳輸層安全

- **強制使用 HTTPS**（TLS 1.2 以上）
- **僅允許各業者提供之白名單 IP 呼叫**

## API 認證

所有 API 請求必須包含以下 Headers：

```
X-VASP-ID: {vasp_id}
X-Timestamp: {ISO8601_timestamp}
X-Signature: {request_signature}
X-Nonce: {unique_random_string_32_chars}
```

<Accordion title="[v2.0 新增] X-Nonce 防重放攻擊機制">
每個 API 請求必須包含一個唯一的 32 字元隨機字串作為 Nonce。接收方 VASP 應：

1. 檢查該 Nonce 是否已被使用過（建議保留至少 24 小時的 Nonce 記錄）
2. 若 Nonce 重複，回傳 `409 Conflict` 錯誤（錯誤代碼 `DUPLICATE_NONCE`）
3. 將 Nonce 納入簽章計算，確保攻擊者無法竄改 Nonce 值

**威脅情境**：攻擊者可能攔截合法的 `/transfer` 請求，在簽章過期前重複發送相同請求，可能造成重複交易或資料洩漏。Nonce 機制可有效防止此類重放攻擊。
</Accordion>

### 簽章計算方式

```
signature = HMAC-SHA256(
  key: shared_secret,
  message: "{method}\n{path}\n{timestamp}\n{nonce}\n{body_hash}"
)
body_hash = SHA256(request_body)
```

<Note>
**[v2.0 變更]** 簽章計算中新增 `{nonce}` 欄位，確保 Nonce 與簽章綁定，防止攻擊者替換 Nonce 後重送請求。
</Note>

### shared_secret 共享機制

<Note>
**[v2.2 新增] 第四次技術會議決議（提案 15）**

`shared_secret` 用於 HMAC-SHA256 簽章計算。由於 mTLS 已於 v2.2 自規格移除（見變更紀錄），`shared_secret` 為目前**唯一的 VASP 身分認證機制**，各家應審慎保管。

**Phase 1 共享方式**：由**公會秘書處統一產生並分發**。

- 秘書處為每一對互通的 VASP 產生獨立的 `shared_secret`（pairwise），避免單一密鑰外洩波及全網
- 透過安全管道（如 PGP 加密郵件、面對面）交付
- **輪換頻率**：建議每年輪換一次；任一方懷疑外洩時應立即向秘書處申請重新分發
- `shared_secret` 屬高敏感憑證，不得以明文形式存入程式碼或版本控制系統
</Note>

## 敏感資料加密

發起人 / 受益人的個人資訊（PII）採用 **Hybrid Encryption（RSA-OAEP + AES-256-GCM）** 整包加密為 `private_info` envelope 傳輸，而非逐欄位加密。詳細欄位定義與 pseudo-code 請參見 [POST /transfer](/spec/endpoints/transfer) 與 [資料模型](/spec/data-models)。

- **金鑰封裝（Key Wrap）演算法**：RSA-OAEP（SHA-256 hash + MGF1-SHA-256），用於加密每筆訊息隨機產生的 AES key
- **內容加密（Content Encryption）演算法**：AES-256-GCM（96-bit IV、128-bit auth tag）
- **加密範圍**：包含 `originator` 與 `beneficiary` 完整物件（含 name、identification、date_of_birth、place_of_birth、physical_address 等所有 PII 欄位）

<Note>
**[v2.1.1 變更]** 由 v2.0 / v2.1 的「逐欄位 RSA-OAEP 加密」改為「整包 RSA + AES Hybrid Encryption」。動機：(1) RSA-OAEP 有明文長度上限（RSA-2048 約 190 bytes、RSA-4096 約 446 bytes），無法容納結構化 `physical_address` 與未來欄位擴充；(2) AES-256-GCM 對大塊資料效能遠優於 RSA；(3) GCM 的 auth tag 提供 envelope 級完整性驗證。

**[v2.1.1 演算法收斂]** 移除 ECIES 選項，統一採用 RSA-OAEP。允許 RSA-2048 與 RSA-4096，**建議使用 RSA-4096**，避免 RSA-2048 在 2030 年左右強度不足。
</Note>

### 金鑰輪換機制

<Accordion title="[v2.0 新增] VASP 間金鑰輪換流程">
在第一階段的分散式架構下（無中央 Registry），各 VASP 透過以下機制管理金鑰輪換：

1. 每個 VASP 的 `GET /vasp/info` 回應中，`public_keys` 陣列包含公鑰的 `expires_at` 欄位
2. 各 VASP 應定期呼叫其他 VASP 的 `GET /vasp/info`，建議頻率不低於金鑰過期時間的一半（例如：金鑰有效期 1 年，則至少每 6 個月查詢一次）
3. 當偵測到目標 VASP 的公鑰即將過期（建議提前 30 天），應主動取得新公鑰
4. 金鑰輪換期間，VASP 可同時保留新舊公鑰（舊鑰 status 設為 `rotating`，新鑰 status 設為 `active`），確保轉換期間不影響服務

**注意**：此為第一階段簡易做法。後續版本應規劃更完善的金鑰管理機制（如集中式金鑰目錄或憑證機制），避免持續依賴手動傳遞公鑰。
</Accordion>

### revoked 金鑰保留期限

<Note>
**[v2.2 新增] 第四次技術會議決議（提案 3）**

為避免 `public_keys` 清單隨金鑰輪換無限膨脹，`revoked` 狀態金鑰的保留規範如下：

- **對外公開**：`revoked` 金鑰應於 `GET /vasp/info` 的 `public_keys` 陣列中保留**至少 90 天**，供對方 VASP 辨識歷史訊息所用金鑰；超過 90 天後即可自清單移除
- **內部保存**：接收方 VASP 應於本地保存對應的 RSA private key **至少 5 年**，以解密資料保留期間內的歷史 `private_info` envelope
</Note>

## IP 白名單

各 VASP 應維護境內 VASP 的 IP 白名單，僅接受來自已註冊 VASP 的請求。
